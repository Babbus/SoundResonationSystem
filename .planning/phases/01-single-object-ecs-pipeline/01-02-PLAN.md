  <name>Task 1: Create StrikeInputManager MonoBehaviour</name>
  <files>
    Assets/SoundResonance/Runtime/Input/StrikeInputManager.cs
  </files>
  <action>
    Create the Input/ directory under Runtime/ if it does not exist. Create StrikeInputManager.cs -- a MonoBehaviour that bridges mouse clicks to ECS StrikeEvent components.

    **Architecture:** This MonoBehaviour lives OUTSIDE the SubScene (on a GameObject in the main scene hierarchy). It uses the New Input System (`UnityEngine.InputSystem`) for mouse input and queries ECS entities to find the click target.

    **Input detection:** On each Update, check `Mouse.current.leftButton.wasPressedThisFrame`. If true, proceed to entity picking.

    **Screen-space entity picking (recommended approach from research):**
    Since SubScene objects have colliders stripped, use screen-space distance matching:
    1. Get mouse screen position from `Mouse.current.position.ReadValue()`
    2. Create an EntityQuery for `ComponentType.ReadOnly<ResonantObjectData>()` and `ComponentType.ReadOnly<Unity.Transforms.LocalToWorld>()`
    3. Iterate all matching entities using `query.ToEntityArray()` and `query.ToComponentDataArray<LocalToWorld>()`
    4. For each entity, project its `LocalToWorld.Position` to screen space via `Camera.main.WorldToScreenPoint()`
    5. Find the entity whose screen position is closest to the mouse click, within a configurable `pickRadius` (default 50 pixels)
    6. If a match is found, enable StrikeEvent on that entity: `entityManager.SetComponentEnabled<StrikeEvent>(entity, true)` and `entityManager.SetComponentData(entity, new StrikeEvent { NormalizedForce = 1.0f })` (fixed force per CONTEXT.md)

    **Important implementation details:**
    - Cache `EntityManager` reference in `Start()` via `World.DefaultGameObjectInjectionWorld.EntityManager`
    - Cache `Camera.main` reference in `Start()` to avoid per-frame lookup
    - Dispose NativeArrays from query results (use `Allocator.TempJob` or `Allocator.Temp`)
    - The EntityQuery should be created once in `OnCreate` or cached, not every frame
    - Add `[SerializeField] private float pickRadius = 50f;` for configurable screen-space pick distance
    - Add a `using UnityEngine.InputSystem;` for the New Input System
    - Namespace: `SoundResonance`
    - This is NOT Burst-compiled (managed MonoBehaviour) -- that is correct and expected for the input bridge

    **Why NOT Physics.Raycast:** SubScene baking strips standard colliders from entities (Pitfall 5 in research). Screen-space picking avoids this entirely and works for the thesis demo scale of 2-3 objects.
  </action>

    Create a Unity test scene programmatically or via editor scripting. The scene needs:

    **Scene hierarchy:**
    1. Main Camera (default, positioned to see all objects)
    2. Directional Light (default)
    3. **StrikeInputManager** -- empty GameObject with StrikeInputManager component (NOT in SubScene)
    4. **ResonanceTestSubScene** -- a SubScene containing:
       - **SteelBar** -- elongated cube (scale ~(0.1, 0.1, 0.5) for bar shape), with ResonantObjectAuthoring + Steel MaterialProfileSO, MeshFilter (Cube), MeshRenderer
       - **GlassPlate** -- flat cube (scale ~(0.3, 0.01, 0.3) for plate shape), with ResonantObjectAuthoring + Glass MaterialProfileSO, positioned offset from steel bar
       - **WoodBar** -- elongated cube (scale ~(0.08, 0.08, 0.4) for bar shape), with ResonantObjectAuthoring + Wood_Oak MaterialProfileSO, positioned offset

    **Approach for scene creation:**
    Since Unity scenes are binary YAML and cannot be trivially hand-written, create an Editor script `Assets/SoundResonance/Editor/TestSceneSetup.cs` that:
    1. Creates a new scene via `EditorSceneManager.NewScene()`
    2. Creates the GameObjects with correct components and transforms
    3. Assigns material profiles from the existing MaterialDatabase assets (find them via `AssetDatabase`)
    4. Creates a SubScene from the resonant objects
    5. Saves the scene to `Assets/Scenes/ResonanceTestScene.unity`
    6. Accessible via menu: `Sound Resonance/Create Test Scene`

    Alternatively, if creating the SubScene programmatically is too complex, create the scene WITHOUT a SubScene -- place resonant objects directly in the scene. This means baking happens at runtime via the Baker, which still works. The trade-off is slightly slower scene load, which is irrelevant for a 3-object test scene.

    **Simpler alternative (preferred):** Create the scene with objects directly in the scene (no SubScene). Add a note that for production, objects should be in a SubScene. For the thesis demo, this avoids the SubScene collider stripping issue entirely AND allows standard Physics.Raycast as a fallback input method.

    If going with the no-SubScene approach, also add Collider components (BoxCollider) to each resonant object, and update StrikeInputManager to use Physics.Raycast as the primary pick method (simpler, more reliable). Keep screen-space picking as the fallback. The research recommends screen-space picking for SubScene objects, but without a SubScene, standard raycasting is simpler and more robust.

    The MaterialProfileSO assets for Steel, Glass, and Wood_Oak should already exist in the project (created by MaterialPresetGenerator). If they don't exist, the editor script should create them from MaterialDatabase.GetPresetData().
